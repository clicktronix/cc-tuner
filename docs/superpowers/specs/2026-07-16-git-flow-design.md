# `cc-tuner:git-flow` — скилл + setup-команда — дизайн

- **Дата:** 2026-07-16
- **Статус:** утверждено, ждёт имплементации (целевая версия v0.5.0)
- **Где живёт:** плагин **cc-tuner** (`plugins/cc-tuner/skills/git-flow/`, `plugins/cc-tuner/commands/git-flow-setup.md`, `plugins/cc-tuner/assets/git-flow/`)
- **Зависит от:** ничего внешнего; интеграция с `/cc-tuner:execute-task` — опциональная (шаги доски скипаются без конфига)

## 1. Зачем

Канон git-флоу (ветки, Conventional Commits, PR-гейты, доска GitHub Projects, планы) сейчас
существует как `.claude/rules/git-flow.md`, **скопированный в 11 отдельных репозиториев**
воркспейсов marqa/stokli. Копии разъехались на два варианта:

- **marqa** (249 строк, обновлён 2026-07-01, rules-hygiene audit): breaking changes
  (`!` + `BREAKING CHANGE:` футер), спецификация футеров, `docs/PLANS/` + `docs/ARCHIVE/`,
  политика «маленькие атомарные PR — ок».
- **stokli** (233 строки, отстал с ~2026-06-19): без breaking changes, `docs/plans/`,
  **противоположная** политика мелких PR (`no-tiny-doc-prs.md`: батчить 3+).

Плюс сопутствующие проблемы: файл без `paths:` frontmatter грузится в каждую сессию
(~2k токенов), хотя процедурные разделы (доска, планы) нужны эпизодически; политики
противоречат друг другу; обновление = 11 ручных правок.

Это ровно болезнь, ради которой существует cc-tuner (README: claude-md-writer был
скопирован в ~10 папок и разъехался → плагин убивает дрейф).

**Ограничение платформы:** плагины Claude Code не умеют шипить `.claude/rules/*`
(компоненты: skills, commands, agents, hooks, MCP, LSP, monitors, bin, settings —
rules нет). Поэтому доставка правила — командой-инсталлером, по прецеденту
`/cc-tuner:statusline-setup` («плагин не может зарегистрировать statusline → ставим командой»).

## 2. Решения брейншторма (зафиксированы с пользователем 2026-07-16)

| Вопрос | Решение |
| --- | --- |
| Где живёт | В cc-tuner, не отдельный репозиторий |
| Доставка канона в репо | Setup-команда пишет `.claude/rules/git-flow.md` из шаблона плагина, с version-маркером |
| Enforcement-хуки (PreToolUse-блокировки) | **Нет** — все запреты остаются advisory-текстом |
| Управление доской | gh-рецепты в скилле + интеграция в execute-task (без отдельной `/task`-команды) |
| Планы | `wiki/PLANS/` + `wiki/ARCHIVE/PLANS/`; фолбэк `docs/PLANS/` пока репо не мигрировал на `wiki/` |
| Мелкие doc-PR | Канон — stokli-политика: не открывать PR ради одного doc-файла; фолдить в открытый PR или батчить 3+; standalone только urgent с подтверждения пользователя (фидбек 2026-06-05) |

## 3. Компоненты

```
plugins/cc-tuner/
  skills/git-flow/SKILL.md            # процедурное знание (on-demand)
  commands/git-flow-setup.md          # /cc-tuner:git-flow-setup — инсталлер правила
  assets/git-flow/rule.template.md    # канон always-on ядра; version-маркер первой строкой
  commands/execute-task.md            # правки: intake/DoD-интеграция с доской
  tests/scenarios/git-flow/*.json     # эвал-сценарии по конвенции репо
```

Версия правила = версия плагина: маркер `<!-- cc-tuner:git-flow v0.5.0 -->` первой
строкой шаблона.

## 4. Флоу целиком (жизненный цикл задачи)

1. **Рождение задачи.** Issue создаётся сразу на доске со Status/Priority
   (`gh issue create --project "<title>"`; фолбэк `gh project item-add` + `item-edit`
   по рецепту скилла). Deferred-финдинги код-ревью — по одному issue на финдинг, тоже на доску.
2. **Старт.** Ветка `<type>/<issue>-<kebab-slug>`; карточка → In Progress.
   Через execute-task — автоматически на intake; вне его — вручную по правилу.
3. **План** (критерии: >1 PR / >2 дней / 3+ компонент). Черновик скретчится superpowers'ом
   в git-ignored `docs/superpowers/plans/`; стоящий план промоутится в трекаемый
   `wiki/PLANS/YYYY-MM-DD-<slug>.md`. План ссылается на issue в первом абзаце, issue — на план.
4. **Работа.** Conventional Commits v1.0.0 + breaking changes (`!` и/или
   `BREAKING CHANGE:` футер — берём из свежего marqa-варианта); явный стейджинг по путям;
   новый коммит вместо `--amend`; WIP-цепочка ок — squash на merge схлопнет.
5. **PR.** Verification gate: чеклист lint/typecheck/tests с реальным выводом;
   `Closes #N` / `Refs #N`; doc-only мелочь не открывает свой PR (см. §2, tiny-PR политика).
6. **Merge.** Squash в main + delete branch; карточка → Done; завершённый план →
   `wiki/ARCHIVE/PLANS/`; follow-up'ы — новые issue на доску.

## 5. Разделение контента: правило vs скилл

**Правило (always-on, ~70 строк, ставится setup-командой):**

- naming веток + типы;
- формат коммита + breaking changes;
- запреты advisory-текстом: никогда в main, no `--amend` / `--no-verify` / `add -A`,
  no force-push в main;
- tiny-PR батч-политика;
- «PR: verification checklist + issue-линк»;
- путь планов (подставлен setup-командой под layout репо);
- последняя строка — указатель: «процедуры доски, merge-стратегии, планы, анти-паттерны —
  invoke `cc-tuner:git-flow`».

**Скилл (on-demand):**

- merge-стратегии и stacked PRs;
- gh-рецепты доски полностью: создание с `--project`, `field-list` / `item-edit`,
  типовые Status/Priority, известные грабли (`--project` берёт title, не `owner/number`);
- lifecycle планов с шаблоном заголовка (Goal / Architecture / задачи с file paths);
- анти-паттерны с реальными кейсами: ghost-conflicts инцидент (PR #50–61, 2026-06-04-05),
  board-инцидент (9 issue мимо доски, 2026-06-05), amend-after-hook-failure,
  first-words branch name, orphan branch;
- pre-PR чеклист;
- секция «Why» с датами инцидентов.

CSO-описание скилла: триггеры — коммиты, ветки, PR, доска, планы; без пересказа процедуры
(description = when, not workflow).

## 6. Setup-команда `/cc-tuner:git-flow-setup`

1. **Детект layout:** `wiki/` существует → путь планов `wiki/PLANS/`; иначе `docs/PLANS/`
   (+ строка-напоминание о миграции). Подставляется в шаблон при записи.
2. **Существующий `.claude/rules/git-flow.md`:**
   - с нашим маркером → сравнить версию; показать дифф; обновить. Локальные правки поверх
     маркерной версии не затирать молча — показать и спросить;
   - легаси-копия без маркера (текущие 11 файлов) → показать дифф с каноном, заменить
     после подтверждения;
   - файла нет → создать.
3. **Репо-дельты:** канон не редактируется на месте. Локальные отличия (labels, номер
   доски, имя project, локальные исключения) живут в соседнем
   `.claude/rules/git-flow.local.md`; setup создаёт его пустым с комментарием-подсказкой
   и **никогда не трогает** при обновлениях.
4. **Легаси-чистка:** детект `no-tiny-doc-prs.md` → предложить удалить (политика теперь
   в каноне). Ничего не удаляется без подтверждения.
5. **Идемпотентность:** повторный запуск на актуальной версии — «up to date», без записи.

## 7. Интеграция в execute-task

Минимально-инвазивные правки, нумерация шагов не меняется:

- **Step 1 (Intake):** после фетча issue — если tracker GitHub и issue на доске,
  карточка → In Progress. Рецепт — отсылкой к скиллу `git-flow`, gh-команды в
  execute-task не дублируются.
- **DoD (finishing):** три новых пункта чеклиста: PR слинкован (`Closes #N`);
  карточка → Done; план (если был) перемещён в архив. Провал любого — задача не завершена.
- **config.template.md:** новый опциональный ключ `board` (project title + owner).
  Пустой/отсутствует → шаги доски скипаются с записью в журнал.

## 9. Миграция (вне скоупа плагина, после релиза)

Прогнать `/cc-tuner:git-flow-setup` в 11 воркспейсах marqa/stokli:

- stokli-копии догоняют marqa-вариант + новые решения (wiki/PLANS-фолбэк, батч-политика);
- `no-tiny-doc-prs.md` удаляется (суть — в каноне);
- существующие расхождения при необходимости уезжают в `git-flow.local.md`.

## 10. Не делаем (отклонено)

- **PreToolUse-хуки** — пользователь выбрал advisory-only.
- **Отдельная `/task`-команда** — рецепты в скилле + execute-task достаточно.
- **Отдельный репозиторий/плагин** — живёт в cc-tuner.
- **Автоматическая миграция `docs/` → `wiki/`** — setup только детектит layout,
  переименование папок — осознанное решение владельца репо.
- **Глобальное правило `~/.claude/rules/git-flow.md`** — отклонено в пользу
  пер-репо установки (версионируется в git проекта, видно тиммейтам и другим агентам).
