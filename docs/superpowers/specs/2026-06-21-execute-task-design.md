# `/cc-tuner:execute-task` — дизайн

- **Дата:** 2026-06-21
- **Статус:** дизайн (до реализации)
- **Где живёт:** плагин **cc-tuner** (`plugins/cc-tuner/commands/execute-task.md`)
- **Зависит от:** `superpowers`, `cc-codex-triage` — внешние плагины, проверяются в рантайме (в CC plugin.json нет механизма зависимостей)

## 1. Зачем

Один воспроизводимый прогон жизненного цикла задачи: от чтения issue до мержа. Раньше эти 10 шагов делались руками (эталон — арк #78); цель — повторяемость **без потери** трёх вещей, которые ломает любая «полная автоматизация»: твоих человеко-гейтов, судейства (валидация возражений Codex против кода) и адаптивности («ревью нашло баг → починил → всплыл рипл → починил глубже»).

**Почему slash-плейбук, а не Workflow и не авто-скил:**
- Монолитный Workflow исполняется в фоне, headless — не может ни спросить тебя (AskUserQuestion), ни показать UI, ни «передумать». Он годится только для fan-out фаз.
- Авто-триггер-скил может не подгрузиться; задачу запускаешь осознанно.
- → `disable-model-invocation` slash-команда, которую **главный агент** проходит по шагам, дёргая Workflow только там, где fan-out реально выгоден.

## 2. Артефакт

- Файл: `plugins/cc-tuner/commands/execute-task.md`, `disable-model-invocation: true`.
- Вызов: `/cc-tuner:execute-task <issue-ref> [--autonomy <mode>]`.
- cc-tuner расширяется из «настройка конфига» в «персональный тюнинг воркфлоу». Остальные части плагина (statusline, claude-md-writer) остаются самодостаточными.

## 3. Зависимости и prereq-чек

cc-tuner ставится **standalone**; зависимость от superpowers/cc-codex-triage есть **только у этой команды**.

**Шаг 0** — bash проверяет наличие обоих плагинов в `~/.claude/plugins/cache/` по якорным файлам:
- superpowers → `.../superpowers/*/skills/brainstorming/SKILL.md`
- cc-codex-triage → `.../cc-codex-triage/*/commands/review.md`

Чего-то нет → печатает команду доустановки (`/plugin install …`) и **мягкий стоп**. Логики graceful-degradation нет намеренно (YAGNI: у пользователя обе всегда стоят).

## 4. Проектный конфиг `.claude/execute-task.md`

Команда читает его на шаге 0. Спина (последовательность) — generic; всё проектное — в этом файле. Нет файла → команда предлагает создать из шаблона один раз, затем продолжает.

Поля:

| Поле | Что задаёт |
|---|---|
| `test` | как прогнать тесты/смоук, в т.ч. UI (`manual: open <url>` или команда) |
| `ci` | команда CI; ручной ли запуск и чем триггерить (`gh workflow run …` / `glab`) |
| `merge` | политика мержа: squash/merge, target-ветка, `auto` или подтверждение |
| `tracker` | как тянуть issue (`gh` / `glab` / `none`) |
| `dor_dod` | шаблон DoR/DoD и acceptance-критерии; пометка, какие критерии машинно-проверяемы, а какие требуют глаза |
| `review_passes` | какие слои ревью обязательны, какие по риску (дефолт: code-review + Codex всегда; requesting-code-review по риску) |
| `autonomy` | дефолтный режим автономии (см. §5) |

## 5. Уровень автономии

Спрашивается **один раз на старте** (если не задан флагом `--autonomy` или полем `autonomy:` в конфиге).

| Режим | Ты в петле на | Остальное |
|---|---|---|
| `brainstorm-only` | только шаг 1 (brainstorm + приёмка DoR/DoD) | шаги 2–10 автономно |
| `checkpoints` | brainstorm + UI-приёмка + мерж | остальное автономно |
| `supervised` | каждый человеко-гейт (🚦) | — |

**Честная граница — что автономия снимает, а что нет.** Автономия гасит рутинные «посмотри-это» чекпоинты, но не жёсткие стопы:
- **Снимается:** шаги 2 и 7 — агент валидирует возражения Codex сам (он и так обязан проверять находки против кода), просто не показывает каждый раунд.
- **Зависит от критериев:** шаг 4 (приёмка) — если acceptance-критерии машинно-проверяемы (chrome-devtools), авто; если требуют глаза, агент делает скриншот + лучшую программную проверку и либо стопает, либо идёт с пометкой «не верифицировано глазами».
- **Не снимается никогда (hard-stop в любом режиме):** провал prereq (0), красный CI/автогейт (3.5/9), любая развилка, которую агент не может разрешить сам (встаёт и спрашивает — автономия ≠ молча выбрать), и **мерж** (см. ниже).

**Мерж в автономном режиме.** По умолчанию даже в `brainstorm-only` — **один финальный стоп-подтверждение** перед мержем (мерж outward-facing и необратим). Полный zero-touch включается явно: `merge: auto` в конфиге. *(Дефолт выбран здесь; легко флипнуть.)*

## 6. Спина — шаги

| # | Шаг | Механизм | Гейт (supervised) | В `brainstorm-only` |
|---|---|---|---|---|
| 0 | prereq-чек + загрузка `.claude/execute-task.md` + выбор autonomy | bash + read | — | — |
| 1 | читаем issue/план; неясно → brainstorm; пишем DoR/DoD | `superpowers:brainstorming` | 🚦 | **твой** (суть режима) |
| 2 | план → `writing-plans` → `cc-codex-triage:plan` до APPROVE | writing-plans + Codex (судейство) | 🚦 | авто (агент судит сам) |
| 3 | реализация субагентами | `subagent-driven-development` (+ Workflow fan-out если юниты независимы) | — | авто |
| 3.5 | **дешёвый автогейт: types/lint/unit** | bash из `ci`/`test` | авто | авто; красный → hard-stop |
| 4 | смоук по acceptance-критериям (чаще UI) | chrome-devtools / `verify` / ты | 🚦 | авто если критерии машинные, иначе пометка/стоп |
| 5 | `/code-review` (xhigh) → правим | code-review (уже fan-out workflow) | судейство | авто |
| 6 | `requesting-code-review` (субагент) → правим | requesting-code-review | **условно по риску** (`review_passes`) | авто/скип по риску |
| 7 | `cc-codex-triage:review` до APPROVE → правим | Codex (цикл, судейство) | 🚦 | авто (агент судит сам) |
| 7.5 | **ре-смоук, если правки 5–7 трогали FE/поведение** | повтор смоука | авто-условно | авто-условно |
| 8 | сверка с планом, отметки сделанного | главный агент | — | авто |
| 9 | CI/CD (запуск если ручной) | bash/`gh`/`glab` из `ci` | авто; красный → hard-stop | авто; красный → hard-stop |
| 10 | мерж | главный агент | 🚦 | финальный стоп (или авто при `merge: auto`) |

Шаги 3.5 и 7.5 — починки процесса относительно исходных 10 шагов: дешёвый автогейт **до** дорогого ревью-марафона, и замыкание верификации (смоук был только до правок ревью).

## 7. Где зовётся Workflow (только fan-out)

- Шаг 5 — `/code-review` уже воркфлоу (finder-углы + верификация).
- Шаг 3 — опц., если юниты реализации независимы (`pipeline()`/`parallel()`, при параллельной правке файлов — `isolation: 'worktree'`).
- Опц. `/simplify` между 7 и 8.

Всё прочее — оркестрация в главной петле с гейтами.

## 8. Прогресс и возобновление

На старте команда заводит Task-list по шагам 1–10. На каждом 🚦 (в текущем режиме) останавливается и ждёт тебя; после ответа — следующий шаг. Прервалась сессия — Task-list показывает, на каком шаге стояли, продолжаем оттуда.

## 9. Не-цели (out of scope)

- Не оркеструет несколько репозиториев за один прогон.
- Не заменяет существующие скилы — только секвенирует их (`brainstorming`, `writing-plans`, `subagent-driven-development`, `code-review`, `requesting-code-review`, cc-codex-triage `/plan` и `/review`).
- Не объявляет hard-dependency в манифесте (CC этого не умеет) — только runtime prereq-чек.
- Не строит логику graceful-degradation под отсутствие зависимостей.

## 10. Открытые вопросы / на будущее

- Машинная проверяемость acceptance-критериев: как размечать в DoR/DoD, что проверяемо программно (chrome-devtools), а что — глаз.
- Авто-определение, какие правки «трогают FE» для условного шага 7.5 (по путям из diff vs `test`-конфига).
- Возможный `--resume` как явная команда (пока возобновление — через Task-list вручную).
