# `/cc-tuner:execute-task` — дизайн

- **Дата:** 2026-06-21
- **Статус:** дизайн (до реализации)
- **Где живёт:** плагин **cc-tuner** (`plugins/cc-tuner/commands/execute-task.md`)
- **Зависит от:** `superpowers`, `cc-codex-triage` — внешние плагины, проверяются в рантайме (в CC plugin.json нет механизма зависимостей)

## 1. Зачем

Один воспроизводимый прогон жизненного цикла задачи: от чтения issue до мержа. Раньше эти 10 шагов делались руками (эталон — арк #78); цель — повторяемость **без потери** трёх вещей, которые ломает любая «полная автоматизация»: твоих человеко-гейтов, судейства (валидация возражений Codex против кода) и адаптивности («ревью нашло баг → починил → всплыл рипл → починил глубже»).

**Почему slash-плейбук, а не Workflow и не авто-скил:**
- Монолитный Workflow исполняется в фоне, headless — не может ни спросить тебя (AskUserQuestion), ни показать UI, ни «передумать». Он годится только для fan-out фаз.
- Авто-триггер-скил может не подгрузиться; задачу запускаешь осознанно.
- → `disable-model-invocation` slash-команда, которую **главный агент** проходит по шагам, дёргая Workflow только там, где fan-out реально выгоден.

## 2. Артефакт

- Файл: `plugins/cc-tuner/commands/execute-task.md`, `disable-model-invocation: true`.
- Вызов: `/cc-tuner:execute-task <issue-ref> [--autonomy <mode>]`.
- cc-tuner расширяется из «настройка конфига» в «персональный тюнинг воркфлоу». Остальные части плагина (statusline, claude-md-writer) остаются самодостаточными.

## 3. Зависимости и prereq-чек

cc-tuner ставится **standalone**; зависимость от superpowers/cc-codex-triage есть **только у этой команды**.

**Шаг 0** — bash проверяет наличие обоих плагинов в `~/.claude/plugins/cache/` по якорным файлам:
- superpowers → `.../superpowers/*/skills/brainstorming/SKILL.md`
- cc-codex-triage → `.../cc-codex-triage/*/commands/review.md`

Чего-то нет → печатает команду доустановки (`/plugin install …`) и **мягкий стоп**. Логики graceful-degradation нет намеренно (YAGNI: у пользователя обе всегда стоят).

Один якорь на плагин достаточно: скилы/команды плагина ставятся комплектом, так что наличие якоря = наличие всех его частей (`writing-plans`, `subagent-driven-development`, `requesting-code-review` приходят с superpowers; `/plan` — с cc-codex-triage). `code-review`, `verify`, `/simplify` — first-party, проверять не нужно.

## 4. Проектный конфиг `.claude/execute-task.md`

Команда читает его на шаге 0. Спина (последовательность) — generic; всё проектное — в этом файле. Нет файла → команда предлагает создать из шаблона один раз, затем продолжает.

Поля:

| Поле | Что задаёт |
|---|---|
| `test` | полный тест/смоук, в т.ч. UI (`manual: open <url>` или команда) |
| `cheap_gate` | дешёвый автогейт для шага 3.5 (types/lint/unit) — **отдельно** от полного `test` |
| `ci` | CI/проверки (тесты/линт — безопасно); ручной ли запуск и чем триггерить (`gh workflow run …` / `glab`) |
| `cd` | deploy/публикация/миграция (шаг 9b) — outward-facing; пусто = их нет |
| `branch` | политика ветки: создавать ли feature-ветку, имя/паттерн, требовать ли чистое дерево (см. §6 префлайт) |
| `merge` | политика мержа: squash/merge, target-ветка, `auto` (zero-touch) или подтверждение |
| `tracker` | как тянуть issue (`gh` / `glab` / `none`) |
| `dor_dod` | шаблон DoR/DoD и acceptance-критерии; пометка по каждому: машинно-проверяем или **требует глаза** (hard-stop, см. §5) |
| `allow_unverified_manual` | явный opt-in: завершить с невыполненным human-eye критерием (по умолчанию `false`; `merge: auto` его НЕ перебивает) |
| `review_passes` | какие слои ревью обязательны/по риску. Дефолт: code-review + Codex всегда; requesting-code-review — по **конкретным** risk-правилам (diff трогает auth/миграции/публичный API, или > N файлов) |
| `artifacts` | (опц.) переопределить политику артефактов (§4bis) |
| `autonomy` | дефолтный режим автономии (см. §5) |

### §4bis. Политика артефактов (git/секрет-гигиена)

Прогон порождает операционное состояние, которое **не должно попасть в коммит** (сырые логи тестов, твои решения, заметки об откате, ссылки на APPROVE, скриншоты могут нести секреты):

- **Local-only** (gitignore через `.gitignore` или `.git/info/exclude`): run-journal `.claude/execute-task-runs/`, скриншоты/evidence для «требует глаза», сырой вывод тестов/CI.
- **Коммитится** (это проектные/дизайн-артефакты): `.claude/execute-task.md` (проектная конвенция, как Makefile) и планы из `writing-plans` (`docs/plans/**` / спеки).
- **Префлайт (0.7)** гарантирует ignore-покрытие для local-only до первой записи.
- **Перед outward-facing операцией (9b/10):** команда **отказывает, если застейджены операционные артефакты**, и финальный diff-чек **учитывает untracked** (иначе новые файлы пройдут мимо проверки — урок scope-pin из cc-codex-triage). Стейджинг хирургический, без `git add -A`.

## 5. Уровень автономии

Спрашивается **один раз на старте** (если не задан флагом `--autonomy` или полем `autonomy:` в конфиге).

| Режим | Ты в петле на | Остальное |
|---|---|---|
| `brainstorm-only` | только шаг 1 (brainstorm + приёмка DoR/DoD) | шаги 2–10 автономно |
| `checkpoints` | brainstorm + UI-приёмка + мерж | остальное автономно |
| `supervised` | каждый человеко-гейт (🚦) | — |

**Честная граница — что автономия снимает, а что нет.** Автономия гасит рутинные «посмотри-это» чекпоинты, но не жёсткие стопы:
- **Снимается:** шаги 2 и 7 — агент валидирует возражения Codex сам (он и так обязан проверять находки против кода), просто не показывает каждый раунд.
- **Зависит от критериев:** шаг 4 (приёмка) — если acceptance-критерий машинно-проверяем (chrome-devtools), авто. Критерий, помеченный в DoR/DoD как **«требует глаза», — hard-stop в любом режиме** (на шаге 4 и при ре-смоуке 7.5): агент делает скриншот + программную проверку, но завершить не может, пока ты не подтвердил. `merge: auto` это НЕ перебивает. Единственный обход — явный `allow_unverified_manual: true` (не уровень автономии).
- **Не снимается режимом автономии (hard-stop в любом режиме):** провал prereq (0); красный дешёвый автогейт (3.5) или CI (9a); невыполненный human-eye критерий (4/7.5); любая развилка, которую агент не может разрешить сам (встаёт и спрашивает — автономия ≠ молча выбрать); и любая **outward-facing операция с побочными эффектами** — `cd`/deploy/публикация/миграция (9b) и **мерж** (10).
- **Единственное явное исключение — `merge: auto`:** оно (и только оно) снимает финальный стоп на шаге 10. Оно НЕ снимает остальные hard-stop'ы: при невыполненном human-eye критерии, красном гейте или непройденном 9b мерж всё равно блокируется.

**Мерж в автономном режиме.** По умолчанию даже в `brainstorm-only` — **один финальный стоп-подтверждение** перед мержем (мерж outward-facing и необратим). Полный zero-touch включается явно: `merge: auto` в конфиге. *(Дефолт выбран здесь; легко флипнуть.)*

## 6. Спина — шаги

| # | Шаг | Механизм | Гейт (supervised) | В `brainstorm-only` |
|---|---|---|---|---|
| 0 | prereq-чек + загрузка `.claude/execute-task.md` + выбор autonomy | bash + read | — | — |
| 0.7 | **префлайт:** чистое дерево (по `branch`-политике), создать/подтвердить feature-ветку, записать target-ветку + base SHA, гарантировать ignore-покрытие local-only артефактов (§4bis), завести run-journal `.claude/execute-task-runs/<id>.md` | bash + git | 🚦 если дерево грязное и политика требует решения | hard-stop если грязное дерево не разрешено политикой |
| 1 | читаем issue/план; неясно → brainstorm; пишем DoR/DoD | `superpowers:brainstorming` | 🚦 | **твой** (суть режима) |
| 2 | план → `writing-plans` → `cc-codex-triage:plan` до APPROVE | writing-plans + Codex (судейство) | 🚦 | авто (агент судит сам) |
| 3 | реализация субагентами | `subagent-driven-development` (+ Workflow fan-out если юниты независимы) | — | авто |
| 3.5 | **дешёвый автогейт: types/lint/unit** | bash из `cheap_gate` | авто | авто; красный → hard-stop |
| 4 | смоук по acceptance-критериям (чаще UI) | chrome-devtools / `verify` / ты | 🚦 | авто если критерии машинные, иначе пометка/стоп |
| 5 | `/code-review` (xhigh) → правим | code-review (уже fan-out workflow) | судейство | авто |
| 6 | `requesting-code-review` (субагент) → правим | requesting-code-review | **условно по риску** (`review_passes`) | авто/скип по риску |
| 7 | `cc-codex-triage:review` до APPROVE → правим | Codex (цикл, судейство) | 🚦 | авто (агент судит сам) |
| 7.5 | **ре-смоук, если правки 5–7 трогали FE/поведение** | повтор смоука | авто-условно | авто-условно |
| 8 | сверка с планом, отметки сделанного | главный агент | — | авто |
| 9a | CI/проверки (тесты/линт — безопасно, запуск если ручной) | bash/`gh`/`glab` из `ci` | авто; красный → hard-stop | авто; красный → hard-stop |
| 9b | CD/deploy/публикация/миграция (если есть) | bash из `cd` | 🚦 | **hard-stop** (outward-facing, в любом режиме) |
| 10 | мерж | главный агент | 🚦 | финальный стоп (или авто при `merge: auto`) |

Починки относительно исходных 10 шагов: **0.7** — префлайт (чистое дерево, feature-ветка, base SHA, run-journal) до автономных правок; **3.5** — дешёвый автогейт *до* дорогого ревью-марафона; **7.5** — замыкание верификации (смоук был только до правок ревью); **9a/9b** — разделение безопасного CI и необратимого CD (CD — outward-facing hard-stop до мержа, а не «один стоп на мерже»).

## 7. Где зовётся Workflow (только fan-out)

- Шаг 5 — `/code-review` уже воркфлоу (finder-углы + верификация).
- Шаг 3 — опц., если юниты реализации независимы (`pipeline()`/`parallel()`, при параллельной правке файлов — `isolation: 'worktree'`).
- Опц. `/simplify` между 7 и 8.

Всё прочее — оркестрация в главной петле с гейтами.

## 8. Прогресс и возобновление

Два уровня:
- **Task-list** (в сессии) — шаги 1–10, видно где гейт. На каждом 🚦 команда стопает и ждёт тебя; после ответа — следующий шаг.
- **Run-journal** `.claude/execute-task-runs/<id>.md` (между сессиями, переживает `/compact`) — заводится в префлайте (0.7), каждый шаг дописывает достаточно для безопасного возобновления: ветка, base SHA, путь к плану, режим автономии, вердикты гейтов и ссылки на APPROVE, вывод тестов, нерешённые человеко-решения, заметки об откате/side-effects. Task-list эфемерен; журнал — источник истины при возобновлении (особенно в автономном прогоне, который может пережить компакт).

Перед любой outward-facing операцией (9b CD, 10 мерж) команда показывает точный commit/diff (с untracked, §4bis), классификацию side-effects и путь отката, отказывает при застейдженных операционных артефактах, и пишет это в журнал.

## 9. Не-цели (out of scope)

- Не оркеструет несколько репозиториев за один прогон.
- Не заменяет существующие скилы — только секвенирует их (`brainstorming`, `writing-plans`, `subagent-driven-development`, `code-review`, `requesting-code-review`, cc-codex-triage `/plan` и `/review`).
- Не объявляет hard-dependency в манифесте (CC этого не умеет) — только runtime prereq-чек.
- Не строит логику graceful-degradation под отсутствие зависимостей.

## 10. Открытые вопросы / на будущее

- Точный синтаксис разметки acceptance-критериев в DoR/DoD (что машинно-проверяемо chrome-devtools, что «требует глаза») — поле есть (§4), формат уточняется на этапе плана.
- Авто-определение, какие правки «трогают FE/поведение» для условного шага 7.5 (по путям из diff vs `test`-конфига).
- Конкретные risk-правила для условного прохода 6 (порог N файлов; список чувствительных путей) — заданы как идея в §4, точные значения на этапе плана.
- Возможный явный `--resume <id>` поверх run-journal (пока возобновление — чтение журнала вручную).
